# 小程序实施路线图

## 目标

把当前“可本地演示、部分功能可调用云函数”的版本，完善为“两位真实微信用户可以在不同设备上稳定绑定、同步、长期使用并可提交审核”的版本。

## 当前完成度概览

| 模块 | 当前状态 | 主要缺口 |
| --- | --- | --- |
| 首页、功能入口、个人页 | 界面和交互基本完成 | 首页部分数据仍来自 mock 或本地内存 |
| 微信登录 | 已有 `login`、`upsertCurrentUser` 云函数 | 云端用户 ID 设计会导致不同用户都可能使用 `me` |
| 情侣绑定 | 已有邀请码相关云函数 | 受用户 ID 冲突影响，真实双人绑定可能失败；缺少解绑 |
| 信箱 | 已有本地与云端读写、草稿、已读、定时信界面 | 图片是临时路径；订阅消息未配置；定时信没有云端定时投递 |
| 朋友圈/相册 | 已有动态、草稿、评论、足迹联动云函数 | 图片是临时路径；缺少删除/编辑等完整管理能力 |
| 待办 | 已有云端 CRUD | 需要双设备联调、错误态和并发验证 |
| 每日问答 | 已有云端记录和 AI 分析 | 需要配置 AI、控制成本、补充失败重试与内容安全 |
| 想你信号 | 已有云端发送、历史、已读 | 需要真实双设备联调和通知策略 |
| 足迹 | 已有部分云端能力 | 地图仍是占位展示；位置和媒体能力需完善 |
| 纪念日 | 本地内存可新增和展示 | 没有云端持久化、编辑、删除、提醒 |
| 每日状态/心情 | 本地内存可用 | 没有云端同步；统计仅基于本地数据 |
| 100 件事 | 本地内存可打卡 | 没有云端同步、编辑或共同操作记录 |
| 记忆胶囊 | 本地内存可创建和打开 | 使用硬编码日期；没有云端同步和可靠锁定 |
| 隐私设置 | 入口存在 | 页面与实际能力未实现 |

## P0：先打通真实双人使用

以下任务完成前，不建议继续扩展新功能。

### 1. 重构真实用户身份（代码修复完成，等待云端部署与双真机验收）

当前风险：

- `mock/users.js` 使用固定 `userId: "me"` 和 `userId: "partner"`。
- `services/auth.js` 登录同步时会把当前本地用户的 `userId` 发给 `upsertCurrentUser`。
- 两个真实微信用户首次登录时，都可能在云端创建成 `userId: "me"`。
- `bindCoupleByInviteCode` 会因此把两个人判断成同一个用户，导致绑定失败或数据串联。

实施要求：

- 云端首次创建用户时，由云函数根据 `openid` 生成全局唯一且不可变的 `userId`。
- 小程序端不再向云端指定真实用户 ID。
- 登录成功后，将云端返回的真实 `userId` 写入 session，并更新当前用户映射。
- `me` 和 `partner` 只能作为界面角色概念，不能作为云数据库主身份。
- 检查所有云函数，统一以 `openid -> users -> userId/coupleId` 作为鉴权入口。

验收标准：

- 两个从未登录过的新微信账号可以完成登录、生成邀请码、绑定。
- 双方云端 `userId` 不同，且刷新、重启后保持不变。
- 绑定后双方都能正确识别自己和伴侣。

### 2. 建立云数据库结构、权限与索引

需要正式定义并创建：

- `users`
- `couples`
- `letters`
- `posts`
- `todos`
- `dailyQuestions`
- `missSignals`
- `footprints`
- 后续新增的 `anniversaries`、`statusRecords`、`bucketListItems`、`capsules`

实施要求：

- 为每个集合整理字段、必填项、状态枚举、时间字段和数据归属。
- 建立查询所需索引，重点覆盖 `openId`、`userId`、`coupleId`、`status`、`createdAt`、`updatedAt`、`date`、`letterId`、`postId` 等组合查询。
- 数据库权限默认禁止客户端直接读写，业务操作统一经过云函数。
- 云函数必须验证当前用户确实属于目标 `coupleId`，不能只相信客户端传入的 ID。
- 情侣绑定等多记录写入操作使用事务或补偿机制，避免只更新一半。

验收标准：

- 未绑定用户无法读取或修改其他情侣的数据。
- 客户端伪造 `coupleId`、`userId`、`letterId` 时云函数会拒绝。
- 常用列表查询不依赖全表扫描。

### 3. 部署并联调现有云函数

实施要求：

- 在微信开发者工具中部署全部实际使用的云函数。
- 为云函数固定可控的 `wx-server-sdk` 版本，避免长期使用 `latest`。
- 建立开发环境和生产环境，避免测试数据进入生产集合。
- 逐个验证登录、绑定、信箱、动态、待办、问答、想你信号、足迹。
- 给页面补充明确的加载中、失败、重试、空数据状态。

验收标准：

- 两台真机完成核心链路测试。
- 断网、云函数失败、重复点击时不会产生重复数据或错误成功提示。

### 4. 实现云存储图片上传

当前选择的头像、动态图片和信件图片都是 `tempFilePaths`，只在当前设备临时有效，上传到数据库后另一台设备无法访问。

实施要求：

- 选择图片后使用 `wx.cloud.uploadFile` 上传到云存储。
- 数据库只保存 `fileID` 或稳定可访问地址。
- 覆盖头像、动态/相册、信件、纪念日封面、足迹图片等场景。
- 删除业务数据时处理无引用文件清理。
- 增加图片大小、数量、格式限制和上传失败重试。

验收标准：

- 用户 A 上传的图片，用户 B 在另一台设备和重新登录后仍能查看。

## P1：完成核心产品闭环

### 5. 修复订阅消息设计

当前问题：

- 模板 ID 仍是 `REPLACE_WITH_LETTER_NOTICE_TEMPLATE_ID`。
- 当前主要在发信人编辑信件时请求订阅授权，但订阅授权属于当前操作用户，不能替另一方授权。
- `sendLetterSubscribeNotice` 接收客户端传入的 `letterId` 和 `toUserId`，需要加强调用者与信件归属校验。

实施要求：

- 在微信公众平台申请并配置真实订阅消息模板。
- 在接收方自己的操作场景中引导其订阅，例如信箱设置或通知设置。
- 由云函数根据当前用户和信件关系确定接收方，避免信任客户端传入的 `toUserId`。
- 记录授权、发送、拒绝、失败状态，并提供可理解的 UI。

验收标准：

- 接收方授权后，另一方发送新信时能收到订阅消息。
- 未授权时应用内功能正常，不出现误导性“已通知”提示。

### 6. 实现真正的定时信投递

当前 `sendLetterScheduled` 只保存 `visibleAt`。没有云端定时任务负责到点更新状态或发送提醒。

实施要求：

- 增加定时触发云函数，定期查询到期且未投递的信件。
- 到点后原子更新投递状态，并按授权情况发送订阅消息。
- 防止重复投递，记录投递尝试、成功时间和失败原因。
- 列表中不能提前泄露定时信正文和标题。

验收标准：

- 发信人关闭小程序后，定时信仍能按时变为可读。
- 定时任务重复执行不会重复发送通知。

### 7. 云端化仍为本地内存的核心模块

建议优先级：

1. 纪念日：共同维护、编辑、删除、提醒。
2. 每日状态/心情：双方同步、历史趋势、首页展示。
3. 100 件事：共同打卡、完成者和完成时间。
4. 记忆胶囊：可靠的开启时间和双方同步。

需要特别修复：

- `services/capsule.js` 使用硬编码 `TODAY = "2026-04-23"`。
- `services/companion.js` 中有硬编码 `dateKey: "2026-04-23"`。
- 本地状态位于 `services/local-state.js`，小程序重启后不会持久保留。

验收标准：

- 上述模块的数据在重启、换设备和双方账号之间保持一致。

### 8. 补齐内容管理能力

建议补充：

- 动态：编辑、删除、删除评论、分页加载。
- 信件：发送前确认、草稿自动保存提示、可选的删除/归档策略。
- 纪念日：编辑、删除、封面上传、关联动态与照片。
- 足迹：编辑、删除、真实地图展示、从动态反向查看。
- 100 件事：自定义项目、恢复完成状态、完成记录。

## P2：可靠性、安全与体验

### 9. 修复时间与时区策略

实施要求：

- 统一规定数据库保存 UTC ISO 时间，展示时转换为用户时区。
- 每日问答、今日状态、定时信和纪念日明确使用哪个时区。
- 删除业务逻辑中的硬编码日期。
- 验证跨午夜、夏令时、设备时间错误和不同时区场景。

### 10. 增加安全与内容治理

实施要求：

- 所有写操作限制字段长度、数组数量和允许状态。
- 为图片和用户输入接入微信内容安全能力。
- AI 分析前过滤敏感内容，并限制调用频率与成本。
- 订阅消息、评论、想你信号增加频率限制，防止滥用。
- 日志中避免输出 openid、私密信件正文等敏感信息。

### 11. 完成账号与隐私能力

当前隐私入口只提示“即将上线”。

需要实现：

- 隐私政策与用户协议。
- 数据收集和用途说明。
- 位置、相册、订阅消息等授权说明与拒绝后的降级体验。
- 解绑情侣关系。
- 退出登录或重置本地状态。
- 注销账号和删除个人/情侣数据的规则。
- 提交审核前确认微信平台最新隐私与类目要求。

### 12. 建立测试与发布流程

当前没有自动化测试或统一脚本。

建议至少建立：

- 服务层单元测试：日期、纪念日、待办、信件可见性、身份映射。
- 云函数测试：鉴权、绑定并发、越权访问、重复提交。
- 两账号真机回归清单。
- 开发/体验/生产环境发布清单。
- 云数据库备份和数据迁移方案。

## P3：清理与长期维护

- 清理或确认未注册页面：`pages/features`、`pages/function`、`pages/index`、`pages/memories`、`pages/tools`、`pages/us`。
- 清理未使用服务：`services/album.js`、`services/daily-status.js`、`services/home.js`、`services/memory.js`。
- 合并重复概念，例如 `question.js` 与 `question-daily.js`、`daily-status.js` 与 `dailyMood.js`。
- 为云函数抽取统一的用户鉴权、情侣关系校验、错误码和数据校验逻辑。
- 增加 README、数据库字段文档、部署说明和故障排查说明。

## 推荐实施顺序

### 里程碑 A：两个人能真实绑定

- 修复唯一用户身份。
- 创建数据库集合、权限和索引。
- 部署登录与绑定云函数。
- 两台真机完成绑定测试。

### 里程碑 B：核心互动能跨设备同步

- 接入云存储图片。
- 联通信箱、动态、待办、问答、想你信号。
- 修复订阅消息授权与发送。
- 完成定时信云端投递。

### 里程碑 C：主要功能不再依赖 mock

- 云端化纪念日、状态、100 件事、胶囊。
- 移除硬编码日期。
- 补齐编辑、删除、分页和错误恢复。

### 里程碑 D：可提交审核

- 完成隐私、协议、解绑、注销和内容安全。
- 完成测试、真机回归和生产环境配置。
- 清理废弃页面与代码，准备审核材料。

## 下一步最值得立即做的任务

先修复“真实用户唯一身份”并完成两台真机绑定。这是目前所有双人云端功能的共同前置条件，也是最可能阻塞后续联调的问题。
